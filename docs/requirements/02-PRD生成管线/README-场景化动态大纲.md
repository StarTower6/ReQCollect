# 02-PRD生成管线 — 迭代需求：场景化动态大纲

> **基于** docs/requirements/02-PRD生成管线/README.md 的需求
> **版本**：迭代需求 v1.0
> **日期**：2026-06-08

## 迭代背景

现有 PRD 生成管线固定输出 9 章节，无法适应不同场景的需求。本项目迭代的目标是让 PRD 大纲**根据场景动态调整**，由 AI 自主决策章节结构，用户可干预确认。

## 需求概述

将当前固定的 9 章 Planner 改为**场景自适应 + 用户可干预**的模式。AI 在挖需阶段结束后，根据已收集的需求画像自动判断场景类别并拟定大纲，经用户确认后按大纲生成。

## 功能需求

### 场景基线与模型自主决策（P0）

AI 识别以下三类场景作为基线起点，但**不限于**固定搭配：

| 场景 | 典型特征 | 基础大纲（基线） |
|------|---------|----------------|
| **全量新建** | 从零开始的新系统/新模块 | 全 9 章：背景→角色→流程→功能需求→系统集成→非功能→数据→约束→验收 |
| **迭代** | 已有系统或上一版 PRD 为基础 | 聚焦变更：迭代背景与范围 → 功能需求（新增/修改）→ 变更对比 → 系统集成（如需）→ 验收标准 |
| **轻量级** | 小功能、工具开发、简单需求 | 简化大纲：需求说明 → 功能描述 → 验收标准 |

AI 应根据实际挖到的需求内容**自主调整大纲**：
- 需求涉及多个系统对接 → 即使属轻量级也应自动加入"系统集成"章节
- 新建项目没有数据量约束 → 可去掉"数据需求"章节
- 用户提到合规要求 → 自动补充"非功能需求"章节
- 某章节信息不足（如无角色信息） → 合并或跳过

### 用户确认流程（P0）

1. **拟大纲** — 生成前 AI 展示拟用大纲（章节列表 + 每章简要说明）
2. **用户干预** — 用户可增/删/改章节
3. **确认** — 用户确认后按确认的大纲生成 PRD
4. **生成后可调整** — 生成中如发现遗漏或用户补充信息，可临时追加章节

### 技术要点

- Planner 需改为**先分析画像 → 再决策章节**，而非遍历固定章节列表
- 每个章节应有**条件规则**（什么条件下包含/排除）作为 LLM 决策参考，但最终决定权交给 LLM
- 用户确认的章节列表需保存到会话上下文中，重新生成时复用

## 验收标准

- [ ] 用户完成挖需后，AI 自动判断场景并展示拟用大纲
- [ ] 大纲支持用户增/删/改章节
- [ ] 用户确认后按实际大纲生成 PRD，不产出超出大纲的章节
- [ ] 画像内容变化时（如中途补充了系统集成信息），AI 能自动追加对应章节提案
- [ ] 重新生成（re-Generate）时复用上次确认的大纲
