# 02 — PRD 生成管线

## 概述

基于需求画像，自动生成结构化需求文档（PRD），支持全量生成和增量生成两种模式。

## 当前状态 ✅ 已实现

### 生成管线
- [x] **Planner** — 基于需求画像规划 9 章节
- [x] **Generator** — LLM 逐章流式生成
- [x] **Assembler** — Markdown 全文拼装

### PRD 模板（9 章，企业 IT 方向）
- [x] 1. 项目背景与目标
- [x] 2. 用户角色与用例（含权限矩阵）
- [x] 3. 业务流程（主流程、分支、异常）
- [x] 4. 功能需求（P0/P1 优先级 + 验收条件）
- [x] 5. 系统集成（对接系统、接口、数据流向）
- [x] 6. 非功能需求（性能、安全、合规）
- [x] 7. 数据需求（量级、存储、报表）
- [x] 8. 实施约束与风险
- [x] 9. 验收标准

### 生成模式
- [x] **one_shot** — 全量生成
- [x] **incremental** — 逐章生成，每章需确认后继续（awaiting_approval 事件）

### 文档质量

#### 内容规范（P0）

- [ ] **去技术化描述** — PRD 各章节聚焦业务功能需求，不出现表结构、接口规范、SQL、配置参数等技术术语
- [ ] **业务视角** — 数据需求章节描述业务数据（"每天产生多少条记录"、"按什么维度统计"），不写数据字典；系统集成章节描述"什么系统之间传什么数据"，不写 API 协议
- [ ] **业务方可验收** — 验收标准从业务视角编写，业务方可以直接判断"做完了还是没做"
- [ ] **一致性检查** — PRD 各章节间交叉引用一致性校验（术语、角色名称）
- [ ] **完整度检查** — 画像中的字段是否全部覆盖到 PRD 各章节
- [ ] **格式检查** — Markdown 格式规范、标题层级、列表对齐

### 版本管理
- [ ] **版本历史** — 每次生成的 PRD 自动归档，支持版本对比（diff）
- [ ] **修订模式** — 用户可针对某章节修改，生成 v2 版本而非全量重来
- [ ] **变更记录** — 自动生成 Change Log：v1→v2 改了什么

### 场景化动态大纲（P0）

当前 Planner 固定输出 9 章，需要改为**场景自适应 + 用户可干预**的模式：

#### 场景基线与模型自主决策

AI 在挖需阶段（Phase 1）结束后，根据已收集的需求画像自动判断场景类别：

| 场景 | 典型特征 | 基础大纲（基线） |
|------|---------|----------------|
| **全量新建** | 从零开始的新系统/新模块 | 全 9 章：背景→角色→流程→功能需求→系统集成→非功能→数据→约束→验收 |
| **迭代** | 已有系统或上一版 PRD 为基础 | 聚焦变更：迭代背景与范围 → 功能需求（新增/修改）→ 变更对比 → 系统集成（如需）→ 验收标准 |
| **轻量级** | 小功能、工具开发、简单需求 | 简化大纲：需求说明 → 功能描述 → 验收标准 |

但**不限于**以上固定搭配。AI 应当根据实际挖到的需求内容**自主调整大纲**：
- 如果需求涉及多个系统对接，即使场景是"轻量级"，也应自动加入"系统集成"章节
- 如果新建项目没有数据量约束，可以去掉"数据需求"章节
- 如果用户特别提到合规要求，自动补充"非功能需求"章节
- 如果某个章节信息不足（没有角色信息），AI 可以合并或跳过

#### 用户确认流程

1. **拟大纲** — 生成前，AI 向用户展示拟用大纲（章节列表 + 每章简要说明）
2. **用户干预** — 用户可以增/删/改章节
3. **确认** — 用户确认后，AI 按确认的大纲生成 PRD
4. **生成后还可调整** — 如果生成过程中发现遗漏或用户中途补充，可临时追加章节

#### 技术要点

- Planner 需改为**先分析画像 → 再决策章节**，而不是遍历固定列表
- 每个章节应有**条件规则**（什么条件下包含/排除）作为 LLM 决策的参考，但最终决定权交给 LLM
- 用户确认的章节列表需保存到会话上下文中，re-Generate 时复用
