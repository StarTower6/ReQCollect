# 07 — 导入分析、文档上传与知识库对接

## 概述

PM 通过企业微信与业务方沟通后，积累了大量的会议纪要、聊天记录等非结构化 Markdown 素材。同时，已有项目的需求文档（PRD）需要作为新迭代的输入。本功能为 PM 提供两条能力闭环：

```
企业微信会议记录(MD) ──→ 导入分析 ──→ 提取需求点填充画像
       历史PRD(MD) ───→ 上传作为上下文 ──→ 查缺补漏/生成迭代版本
                                ↓
                       AI 辅助产出 PRD 迭代版本
```

> 🔴 知识库智能对接本期暂不实现，保留为远期规划。

---

## 一、企业微信会议/聊天记录导入分析

### 1.1 业务场景

PM 与业务方通过企业微信（企业微信）沟通需求后，产生以下形式的 Markdown 素材：
- **工作会议纪要** — PM 整理的企业微信会议记录，导出为 Markdown 文档，记录了业务方提出的需求、痛点、期望改动
- **企业微信群聊/单聊记录** — 从企业微信导出的 Markdown 格式聊天记录，包含多轮需求讨论
- **会议录音转文字** — 转录工具输出的 Markdown 文本

PM 希望将这些素材零成本导入系统，AI 自动理解并提取结构化需求字段。

### 1.2 功能需求

#### P0 — 导入入口（核心 MVP）

| # | 需求 | 验收标准 |
|---|------|----------|
| 1.1 | **新建会话时上传记录** | 对话界面提供「导入记录」按钮，弹窗支持选择本地 `.md` 文件，上传后创建一个新会话 |
| 2.1 | **对话中上传** | 在已有对话的消息输入框旁提供「📎 上传」按钮，选择 `.md` 文件后内容作为上下文注入当前对话，AI 参考记录继续追问 |
| 3.1 | **导入到已有会话（入口）** | 在已有会话的侧栏或菜单中提供「导入记录」选项，记录内容追加到当前对话上下文 |

#### P0 — 智能分析（核心 MVP）

| # | 需求 | 验收标准 |
|---|------|----------|
| 4.1 | **自动提取需求画像字段** | 上传记录后，AI 自动扫描内容，从 11 个画像字段中提取已知信息填充到当前会话的画像中 |
| 5.1 | **标记信息来源** | 每个已填充的画像字段旁显示「来源：导入记录-xxx.md」，方便溯源 |
| 6.1 | **未覆盖字段继续追问** | 提取完毕后，AI 自动总结「已从会议记录中获取 X 项信息，还有 Y 项需确认」，并继续追问缺失字段 |
| 7.1 | **模糊/矛盾内容提示** | 记录中表述不清晰或前后矛盾的地方（如"用户量大"但无具体数字），AI 标记为"待确认"并提示 PM 补充 |

#### P1 — 多记录合并（二期）

| # | 需求 | 验收标准 |
|---|------|----------|
| 8.1 | **多次导入合并** | 同一会话可多次上传不同记录文件，AI 自动合并去重，冲突时提示 PM 确认 |
| 9.1 | **记录对比** | 两份记录对同一需求描述不一致时，AI 并排展示差异点供 PM 选择 |
| 10.1 | **来源追溯** | 每项需求的提出人/会议时间可追溯，便于后续确认 |

### 1.3 技术约束

- 首期仅支持 `.md` 格式，单个文件 ≤ 10MB
- 解析基于 LLM 结构化提取（非正则规则），保留原始文本引用
- 上传文件保存在 `pm_data/imports/{session_id}/` 目录

## 二、历史需求文档上传（二期）

> 🔴 **本期核心 MVP 暂不实现** — 以下内容作为二期规划参考

### 2.1 业务场景

PM 在迭代已有项目时，需要参考之前产出的 PRD：
- **同一项目的旧版 PRD** — 在一期基础上做二期，需了解一期范围
- **同类项目的 PRD** — 作为参考样例，帮助结构化思考
- **外部输入的文档** — 其他部门/厂商提供的 Word/PDF 需求文档（首期支持 MD）

### 2.2 功能需求

#### P1 — 上传与存储（二期）

| # | 需求 | 验收标准 |
|---|------|----------|
| 11.1 | **PRD 上传** | 独立的上传页面或会话中上传，支持 `.md` 文件，上传后自动解析并保存 |
| 12.1 | **文档库管理** | 在界面左侧新增「文档库」面板，以列表展示已上传的所有历史 PRD（标题、日期、项目名称） |
| 13.1 | **文档分类** | 支持为文档打标签（项目名/系统名/部门），支持搜索和筛选 |

#### P1 — 作为对话上下文（二期）

| # | 需求 | 验收标准 |
|---|------|----------|
| 14.1 | **选择文档作为上下文** | 新建会话时可选「关联历史文档」，选择后该文档内容作为系统提示注入 AI |
| 15.1 | **自动识别迭代关系** | 上传文档包含项目名称时，AI 自动识别是否为已有项目的迭代版本，在画像中标注「迭代于：xxx」 |
| 16.1 | **差异分析** | 上传旧版 PRD 后，AI 对比当前对话中已收集的需求，自动列出「已有功能保持不变 / 需要变更 / 新增需求」 |

#### P1 — Word/PDF 转换（后续迭代）

| # | 需求 | 验收标准 |
|---|------|----------|
| 17.1 | **格式转换** | 上传 Word/PDF，后端调用转换工具（pandoc / python-docx / marker-pdf）转为 MD 后分析 |
| 18.1 | **格式提醒** | 上传非 MD 文件时，提示「正在转换为 Markdown，可能需要校对」 |

#### P1 — 文档导出与迭代版本管理

| # | 需求 | 验收标准 |
|---|------|----------|
| 19.1 | **PRD 版本存档** | AI 生成 PRD 后，自动存储在文档库中，与该项目关联 |
| 20.1 | **版本树** | 一个项目下的多次 PRD 生成按时间线展示，可查看 v1→v2 的变更摘要 |
| 21.1 | **基于旧 PRD 重开会话** | 在文档库中点「基于此文档重新讨论」，创建一个继承历史画像的新会话 |

### 2.3 技术约束

- 文档库存储路径：`pm_data/documents/`，元信息存 MySQL
- 文档与知识库（Feature 3）共享一套存储，知识库可配置为文档库的子集
- 首期支持 MD，Word/PDF 转换使用 pandoc 或 marker-pdf

## 三、知识库智能对接

> 🔴 **本期暂不实现** — 以下内容作为远期规划参考，不纳入本次开发范围

### 3.1 业务场景

PM 产出的 PRD 需要反映真实系统的功能边界。仅靠口头沟通容易遗漏已有能力或与现有系统矛盾。知识库作为 AI 的"外脑"，在关键决策点提供参考。

常见知识库类型：
- **Git 代码仓库** — 了解已有功能、数据结构、API 接口
- **历史 PRD 文档库** — 系统已保存的所有需求文档
- **企业内部 Wiki/文档平台** — HtmlHub 托管的文档
- **企业微信聊天记录归档** — 历史沟通记录
- **外部 API** — 第三方系统接口文档

### 3.2 功能需求

#### P0 — 知识库配置与管理

| # | 需求 | 验收标准 |
|---|------|----------|
| 22.1 | **知识库注册** | 设置页面提供「知识库管理」面板，支持添加/编辑/删除知识源 |
| 22.2 | **知识源类型** | 内置支持以下类型：<br> - **本地文档库**：系统内的历史 PRD 文档集合<br> - **Git 仓库**：输入仓库 URL + 分支，自动拉取并索引代码文件<br> - **自定义 API**：配置 URL + Token，按接口规范查询<br> - **本地目录**：指定服务器上某目录作为知识库 |
| 23.1 | **索引状态** | 每个知识源显示索引状态（未索引/索引中/已索引/N 条记录）、最后更新时间 |
| 24.1 | **手动/自动索引** | 支持手动触发索引更新，也支持定时自动同步 |

#### P0 — 关键时机自动检索

| # | 需求 | 验收标准 |
|---|------|----------|
| 25.1 | **了解现有系统时** | 当 AI 需要填写 `existing_systems` 字段或用户提到现有系统时，自动查询知识库中该系统的相关资料 |
| 26.1 | **功能需求冲突检测** | AI 生成功能列表时，对比知识库中已有功能，标记「与现有系统功能 X 重叠」或「现有系统已有类似功能 Y」 |
| 27.1 | **技术术语参考** | PM 提到不确定的术语时（如"我们想对接 SAP"），AI 从知识库检索该系统的接口方式/数据格式 |
| 28.1 | **PRD 生成时的背景补充** | 生成 PRD 各章节时，自动从知识库检索相关信息充实内容（如数据量级参考、性能标准） |

#### P1 — 检索策略

| # | 需求 | 验收标准 |
|---|------|----------|
| 29.1 | **RAG 向量检索** | 知识库支持文本向量化（Embedding），基于语义相似度检索最相关片段 |
| 30.1 | **检索结果溯源** | 每次检索结果展示来源（文件名、行号/段落、URL），PM 可点击查看原文 |
| 31.1 | **可见性控制** | 每次 AI 引用知识库内容时，在消息中展示「📖 来自知识库：[知识源名称]」标记 |
| 32.1 | **手动查询** | 对话中 PM 可主动输入「/kb 查询关键词」快速检索知识库，结果以引用形式插入对话 |

#### P1 — 知识库扩展示例

| 类型 | 配置方式 | 用途示例 |
|------|---------|---------|
| 格力企业微信记录归档 | 指定目录下的 MD 文件 | 回顾某业务线的历史需求沟通过程 |
| Git 代码库 | 输入 `StarTower6/ReQCollect` | 了解 ReQCollect 已有 API 和数据模型 |
| HtmlHub 托管的文档 | 配置 HtmlHub API 端点 | 查询公司内部系统架构文档 |
| 自定义 API | 配置 URL + 请求格式 + Token | 对接格力内部的 CMDB/资产管理接口 |

### 3.3 技术约束

- 向量检索使用 FAISS（轻量，免运维），Embedding 模型使用本地或 API 调用
- 知识库索引存储在文件系统（`pm_data/knowledge/index/`），元信息存 MySQL
- 首期不要求实时同步，支持手动触发更新即可

## 四、UI 交互设计

### 4.1 实现方式

在现有原生 HTML/CSS/JS 单页（`static/index.html`）上扩展，不加 Vue/React 等构建工具：
- 导入弹窗用原生 Modal + 表单
- 文档库面板用侧边栏（左侧会话列表下方或右侧新增面板）
- 文件上传用原生 `<input type="file">` + Fetch API
- 样式继承现有 CSS 设计系统，不加外部 UI 框架

### 4.2 三功能在界面的位置

```
┌─────────────┬──────────────────────┬──────────────┐
│  会话列表    │      对话区           │ 需求画像      │
│   ───────   │                      │  ───────     │
│  [导入]按钮  │  上: 消息列表         │  完整度进度环  │
│  [文档库]   │  下: 输入框           │  字段状态     │
│   ───────   │    [📎 上传] [发送]    │              │
│  会话1      │                      │  知识库引用   │
│  会话2      │                      │  📖 来源标签  │
└─────────────┴──────────────────────┴──────────────┘
```

### 4.2 导入对话框

点击「导入记录」弹出对话框：
- 上传区域：拖拽或点击选择 `.md` 文件
- 导入模式选择：
  - **新建会话**：基于此记录创建新对话
  - **导入到当前会话**：追加到当前对话上下文
- 解析进度条 → 提取结果预览 → 确认导入

## 七、数据模型设计（新增）

```sql
-- 文档库
CREATE TABLE documents (
    id            INT AUTO_INCREMENT PRIMARY KEY,
    session_id    INT,                -- 关联的会话（可为 NULL）
    project_name  VARCHAR(255),       -- 项目名称
    title         VARCHAR(255),       -- 文档标题
    filename      VARCHAR(255),       -- 原始文件名
    file_path     VARCHAR(512),       -- 存储路径
    file_type     VARCHAR(20),        -- md
    content_text  LONGTEXT,           -- 提取的纯文本
    tags          JSON,               -- 标签数组
    version       VARCHAR(50),        -- 版本号（如 v1.0, v2.0）
    created_at    DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 文档标签
CREATE TABLE document_tags (
    id          INT AUTO_INCREMENT PRIMARY KEY,
    doc_id      INT NOT NULL,
    tag         VARCHAR(100) NOT NULL,
    FOREIGN KEY (doc_id) REFERENCES documents(id) ON DELETE CASCADE
);

-- 文档导入记录（溯源）
CREATE TABLE import_records (
    id            INT AUTO_INCREMENT PRIMARY KEY,
    session_id    INT NOT NULL,
    filename      VARCHAR(255),
    file_path     VARCHAR(512),
    fields_filled JSON,               -- 本次导入填充了哪些画像字段
    created_at    DATETIME DEFAULT CURRENT_TIMESTAMP
);
```
