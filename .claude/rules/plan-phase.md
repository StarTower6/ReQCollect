# PGE 三段式 — Plan（计划阶段）

## 目标

在动手写代码之前充分理解任务范围、依赖关系和风险点。

## 必需输出

输出一个实施计划，包含以下内容：

```markdown
## 实施计划

### 1. 任务理解
- 需求来源: docs/requirements/<模块>/README.md
- 核心目标: 一句话概括

### 2. 改动清单
- 新增文件: [...]
- 修改文件: [...]
- 删除文件: [...]

### 3. 数据模型 (如有)
- 新增表/字段

### 4. 验收标准
每一项改动必须有可验证的验收条件，标记 P0/P1:

- [ ] P0: MySQL connection pool 可成功建立连接
- [ ] P0: /api/pm/sessions 返回当前用户的所有会话
- [ ] P1: JSON → MySQL 迁移脚本可一键导入
- [ ] P0: 数据库连接失败时优雅降级到 JSON 文件模式

### 5. 风险与依赖
- 依赖关系
- 注意事项

### 6. 实施步骤 (按顺序)
1. ...
2. ...
```

## 验收标准编写规则

- **可量化** — 避免"界面美观"、"性能好"，用具体指标:
  - ✅ "POST /api/pm/agent 返回 SSE 事件流"
  - ✅ "用户登录后能看到自己的历史会话"
  - ❌ "用户体验好"
- **分层级** — P0 = 不做就不能发布, P1 = 迭代可补
- **可测试** — 每条标准对应一个测试命令或 UI 操作路径, 评估阶段直接按清单打勾
- **有边界** — 明确"不做": "本次不实现用户管理，仅做会话隔离"

## 原则

- **读需求** — 必须先读 `docs/requirements/` 下的对应文档
- **尊重现有架构** — 不要推翻现有结构重写
- **小步提交** — 一个功能一次 commit，不要攒一堆一次性提交
- **先确认后动手** — 把计划展示给用户，等确认后再进入 Generate
- **技能准备** — 如果涉及前端设计，先加载 design / impeccable 技能
